# Real-Web Agent Harness Quality Lab 应用设计方案

## 1. 项目背景

我之前在 B 站做过 WebUI 自动化 Agent 项目，核心目标是用自然语言描述测试任务，让 Agent 理解页面 DOM 与上下文，规划点击、输入、等待、断言等动作，再通过 Playwright 执行，并将动作落盘生成可回放脚本。这个项目在 B 站直播 Web 场景中主要服务于 UI 自动化提效：例如直播间送礼、弹幕发送、关注、问题反馈等链路，减少页面频繁改版后手写 UI 自动化脚本的维护成本。

但如果项目只停留在“让 Agent 点页面、生成脚本”，长期价值会被通用 Browser Agent、Coding Agent、Computer Use 类工具冲击。真正有长期价值的方向不是再做一个普通 Browser Agent Demo，而是建设一套 **Agent Harness**：让 Agent 在可控、可测、可观测、可回放、可归因、可持续优化的环境里运行。

因此，本项目的定位是：

> **Real-Web Agent Harness Quality Lab：面向真实 Web 场景的 Agent 执行、需求到用例生成、评测、Trace、Badcase 回流、安全控制与知识治理平台。**

它不是单纯做“自动点页面”，而是做一个小型的 AI 应用质量工程平台，用来验证 Agent 在企业真实交付流程中的可靠性。

---

## 2. 为什么要用 Agent 做

传统测试自动化通常依赖人工编写固定脚本。它的优点是稳定、可控、可维护；缺点是在业务变化快、页面结构频繁改版、需求描述复杂、测试用例需要快速生成时，维护成本高。

Agent 适合引入的地方不是替代所有测试，而是处理这些传统脚本不擅长的部分：

1. **需求理解**
   从需求文档、PRD、测试说明中提取功能点、边界条件、风险点和测试任务。

2. **任务规划**
   将“用户要完成什么”转为“页面上需要执行哪些步骤”。

3. **动态页面适配**
   当页面结构变化时，Agent 可以基于当前 DOM / 视觉 / 上下文重新规划动作，而不是完全依赖固定 XPath。

4. **测试资产生成**
   生成测试点、测试用例、自动化任务、执行报告、风险摘要和 bug 草稿。

5. **失败归因与数据回流**
   Agent 执行失败后，不只是报错，而是可以归因到 context、planner、executor、assertion、tool、skill、knowledge 或 safety 等不同层。

所以本项目使用 Agent 的核心原因是：

> 企业测试与业务交付中存在大量“半结构化输入 + 动态环境 + 多工具调用 + 需要人机协同判断”的任务，Agent 适合承担规划、生成、执行和辅助归因，但必须放在 Harness 里才能工程化落地。

---

## 3. 为什么不能只用普通 Agent，而要用 Harness

普通 Agent 的目标通常是“完成一个任务”。但企业场景更关心：

* 这个任务是否可控；
* 执行过程是否可追踪；
* 失败原因是否可定位；
* 新版本是否比旧版本更好；
* 工具调用是否越权；
* 知识库是否污染；
* 是否可以回归验证；
* 人工如何介入；
* 结果能否作为质量依据。

因此，本项目不直接做裸 Agent，而是构建 Harness。

Harness 的核心价值是：

```text
普通 Agent：关注任务能不能跑通
Agent Harness：关注任务能否被控制、被评测、被复盘、被优化、被安全治理
```

本项目中的 Harness 至少包含以下模块：

```text
需求输入 / Web 页面 / 本地 Fixture
        ↓
Context Collector
        ↓
Context Builder
        ↓
Knowledge Layer
        ↓
Planner
        ↓
Action Schema Validator
        ↓
Safety Guard
        ↓
Tool / Skill Registry
        ↓
Playwright Executor
        ↓
Assertion Layer
        ↓
Trace Store
        ↓
Evaluator
        ↓
Badcase Feedback
        ↓
Knowledge / Prompt / Skill / Eval Dataset 回流
```

最终目标不是“Agent 成功执行一次”，而是：

> 通过固定评测集、Trace、Badcase、知识治理和安全控制，把 Agent 从一次性能力变成可持续迭代的工程系统。

## 4. 项目核心应用场景设计

本项目设计三个核心应用场景。它们不是彼此独立，而是逐步递进。

---

## 4.1 场景一：Web 用例执行与脚本回放

### 场景定位

这是承接 B 站经历的基础场景。

目标是让 Agent 根据自然语言测试任务，在真实公开 Web 页面或本地可控页面中完成操作，并输出可追踪的执行过程和可复用脚本。

### 示例任务

```text
在 Playwright 文档页面中确认 Locators 章节存在，并验证页面包含 getByRole 相关内容。
```

```text
在本地直播间 fixture 页面中点击礼物面板，选择“小花花”，点击发送，并断言出现发送成功 toast。
```

```text
在本地审批系统 fixture 页面中填写请假申请，提交后断言状态变为“待审批”。
```

### 输入

* 自然语言任务；
* 起始 URL；
* 业务上下文；
* 可用 actions；
* 可用 tools / skills；
* 安全策略；
* 断言要求。

### 输出

* action 序列；
* 每步截图；
* 每步 trace；
* 执行结果；
* 断言结果；
* 错误类型；
* 可选 Playwright 脚本。

### 关键能力

```text
Context Collector
Action Schema
Playwright Executor
Assertion Layer
Trace Store
Script Generator
```

### 价值

这个场景解决的是测试执行层问题：

> 页面改版频繁时，通过 Agent 动态理解页面并生成或执行自动化任务，减少手写 UI 自动化脚本的成本。

### 本项目第一阶段实现范围

第一阶段只实现：

```text
打开网页
→ 采集页面 context
→ Fake Planner 输出动作
→ Executor 执行 assert_text
→ Trace 保存
```

暂时不要求复杂模型规划，也不要求真实业务系统登录。

---

## 4.2 场景二：需求交付到测试任务生成

### 场景定位

这是比“用例执行”更高价值的场景，也是更接近飞书 / 字节业务交付流程的场景。

它不是单纯让 Agent 点页面，而是让 Agent 参与需求交付流程：

```text
需求文档
→ 功能点提取
→ 测试点生成
→ 用例生成
→ 自动化任务生成
→ Agent 执行
→ 报告与风险摘要
```

### 为什么这个场景重要

企业测试流程中，很多成本不在“执行用例”本身，而在：

* 理解需求；
* 判断影响范围；
* 设计测试点；
* 生成测试用例；
* 确定哪些用例适合自动化；
* 执行后整理结果；
* 失败后归因；
* 和研发 / 产品同步风险。

Agent 的价值是把这些半结构化流程串起来。

### 示例需求

```markdown
# 需求：直播间新增粉丝团礼物快捷入口

用户进入直播间后，如果已加入主播粉丝团，则在礼物面板入口旁展示“粉丝团礼物”快捷入口。
点击入口后打开礼物面板，并默认选中粉丝团专属礼物。
如果用户余额不足，则点击发送后需要展示余额不足提示。
如果用户未加入粉丝团，则不展示该快捷入口。
```

### Agent 需要生成

```json
{
  "requirement_points": [
    "已加入粉丝团用户展示快捷入口",
    "未加入粉丝团用户不展示快捷入口",
    "点击快捷入口打开礼物面板",
    "默认选中粉丝团礼物",
    "余额不足时展示提示"
  ],
  "test_points": [
    "入口展示条件校验",
    "入口点击交互校验",
    "默认选中状态校验",
    "余额不足异常提示校验",
    "未加入粉丝团场景校验"
  ],
  "automation_tasks": [
    {
      "task": "进入直播间，模拟已加入粉丝团用户，点击粉丝团礼物快捷入口，断言礼物面板打开并默认选中粉丝团礼物",
      "expected": ["gift_panel_opened", "fan_gift_selected"]
    }
  ],
  "risk_points": [
    "快捷入口可能是无文字 icon，DOM 语义不足",
    "礼物面板可能存在懒加载",
    "toast 消失较快，断言需要及时捕获",
    "用户身份和余额需要前置造数或 mock"
  ]
}
```

### 第一版实现方式

先不接真实飞书 / B 站 / 内部系统。

本项目中可以用本地文件模拟需求输入：

```text

docs/sample_requirements/requirement_approval.md
docs/sample_requirements/requirement_docs_search.md
```

然后实现：

```text
Requirement Parser
→ Test Point Generator
→ Automation Task Generator
→ Eval Case Writer
```

第一版可以先用 LLM 生成结构化 JSON，后续再加规则校验。

### 关键能力

```text
Requirement Parser
Knowledge Layer
Test Case Generator
Automation Task Generator
Eval Case Writer
Trace Store
Human Review
```

### 价值

这个场景对应“业务交付流程 Agent 化”：

> 不只是执行测试，而是把需求理解、测试设计、自动化任务生成和执行报告串起来，模拟 AI 融入提测、测试下发和回归验证流程。

## 4.3 场景三：Agent 能力评测与 Badcase 回流

### 场景定位

这是本项目最重要的主线。

Web 用例执行和需求到用例生成都只是具体应用。真正决定项目质量的是：

> 如何评测 Agent 是否变好了，以及失败后如何回流优化。

### 核心问题

Agent 系统不能靠“跑一次成功”证明有效。必须回答：

* 新 prompt 是否比旧 prompt 好？
* Context v2 是否比 Context v1 更稳定？
* 加 Skill 后成功率是否提升？
* 换模型后 JSON 合法率是否下降？
* 失败到底是模型问题、上下文问题、执行器问题，还是断言问题？
* 修复一个 badcase 后是否引入新的回归问题？
* 知识库更新后是否污染旧任务？

### Evaluator 目标

建立固定任务集，对不同版本的 Agent 能力做回归测试。

### 第一版任务集

```text
eval_cases/
├── docs_cases.jsonl
├── fixture_cases.jsonl
└── requirement_cases.jsonl
```


## 5. 视觉模型与体验类检测设计

### 5.1 是否必须依赖视觉模型

体验类检测通常需要视觉模型，但第一阶段不强依赖。

可以分三层做：

```text
第一层：DOM / URL / Trace 断言
第二层：截图规则检测
第三层：视觉模型断言
```

### 第一层：DOM / URL / Trace 断言

适合验证：

* 文本是否存在；
* 元素是否出现；
* 状态是否变化；
* URL 是否跳转；
* 是否调用了预期 Tool / Skill；
* 是否出现错误状态。

这部分不需要视觉模型。

### 第二层：截图规则检测

适合验证一些简单体验问题：

* loading 是否长时间存在；
* 截图是否为空白；
* 某个区域是否有明显变化；
* 弹窗是否出现；
* toast 区域是否出现新文本。

这部分可以先用截图 + 简单规则，不一定需要 VLM。

### 第三层：视觉模型断言

适合验证：

* 按钮遮挡；
* 文字溢出；
* 页面错位；
* 花屏 / 黑屏 / 白屏；
* toast 是否符合预期；
* 弹窗布局是否异常；
* 移动端适配是否异常；
* 样式是否与预期不符。

### 5.2 本地是否能部署视觉模型

可以部署小型视觉模型，但不建议一开始作为主链路依赖。

Ollama 上的 `qwen2.5vl` 支持 Text + Image 输入，提供 3B、7B、32B、72B 等版本，其中 `qwen2.5vl:3b` 模型大小约 3.2GB，`qwen2.5vl:7b` 约 6.0GB，均标注为 Text/Image 输入。([Ollama][1])

结合当前硬件：

```text
MacBook Air 16GB：
- 可尝试 qwen2.5vl:3b
- 不建议把视觉模型作为日常主链路

Windows RTX 3060 6GB：
- qwen2.5vl:3b 更现实
- qwen2.5vl:7b 可尝试，但 6GB 显存下可能紧张，速度和稳定性不一定好
```

因此项目设计中：

```text
MVP：不依赖视觉模型
P1：保留 VisionAssertionProvider 接口
P2：Windows Ollama 尝试 qwen2.5vl:3b
P3：再做 DOM Assertion vs Vision Assertion 对比实验
```

### 5.3 视觉断言第一版接口

```json
{
  "assertion_type": "vision",
  "image_path": "runs/demo/screenshots/step_3.png",
  "question": "页面中是否出现发送成功的 toast？请只输出 true/false 和一句证据。",
  "expected": true
}
```

输出：

```json
{
  "pass": true,
  "evidence": "页面底部出现了包含发送成功含义的 toast 提示",
  "confidence": 0.78,
  "error_type": null
}
```

### 5.4 视觉断言边界

视觉模型结果不能直接作为强质量结论。必须满足：

* 保存截图证据；
* 输出判断依据；
* 低置信度进入人工复核；
* 关键链路优先使用 DOM / API / Trace 多断言交叉验证；
* 视觉断言主要用于体验类问题和 DOM 难以覆盖的问题。

## 6. 知识库设计：RAG、Agent Wiki 与 Memory 的关系

### 6.1 RAG 的局限

RAG 适合从大量文档中检索相关片段，但在 Agent Harness 中会遇到几个问题：

1. **只检索原文，不一定形成稳定知识**
   RAG 返回的是 chunk，Agent 每次还要重新理解、压缩和推理。

2. **知识容易碎片化**
   同一个业务概念可能散落在多个文档中，检索结果不一定完整。

3. **时效性和污染问题**
   旧文档、废弃规则、过期接口可能仍被检索出来。

4. **难以表达执行经验**
   “某个按钮经常没有 aria-label，需要用 class + 周边文本判断”这类经验不一定存在于原始文档中。

5. **不适合直接存放高价值抽象**
   例如组件别名、业务术语、失败归因规则、Skill 适用条件，更适合结构化维护。

### 6.2 Agent Wiki 是什么

“Agent Wiki”不是一个我能确认的统一标准术语；不同团队可能用它指代不同东西。这里采用一个工程化定义：

> Agent Wiki 是面向 Agent 的结构化知识层，用来存放经过人工或系统归纳后的稳定知识，而不是简单存放原始文档 chunk。

它更像：

```text
RAG：查原始资料
Agent Wiki：存沉淀后的知识卡片
Memory：存历史执行经验
Eval Dataset：验证知识是否有效
```

网上也有类似讨论：RAG 能检索大规模文档，但单靠 RAG 不一定能解决跨会话知识综合和稳定记忆问题；Wiki / Memory 类机制更强调将知识沉淀为可复用层。这个方向仍在快速变化，不能把某一种说法当成行业标准。([DEV Community][2])

### 6.3 本项目采用混合知识架构

本项目不做“RAG vs Agent Wiki 二选一”，而采用混合架构：

```text
Raw Docs
- 需求文档
- 历史测试用例
- 业务说明
- 接口文档
- 组件说明

RAG Index
- 面向原始资料检索
- 用于回答“资料里怎么写”

Agent Wiki
- 业务术语
- 组件语义
- 页面类型
- 测试策略
- 断言策略
- 失败处理规则
- Skill 适用条件

Execution Memory
- 历史 trace
- 历史 badcase
- 成功路径
- 高频失败原因

Eval Dataset
- 固定评测集
- 回归集
- 知识变更验证集
```

### 6.4 Agent Wiki 数据结构

#### 业务术语卡片

```json
{
  "id": "term_gift_flower",
  "type": "business_term",
  "name": "小花花",
  "aliases": ["免费礼物", "小花", "flower gift"],
  "definition": "直播间内的一种免费礼物",
  "related_pages": ["live_room", "gift_panel"],
  "related_actions": ["open_gift_panel", "send_gift"],
  "status": "active",
  "source": ["manual", "badcase_feedback"],
  "version": "2026-04-29"
```

### 6.5 知识调用策略

不同任务调用不同知识：

```text
需求到用例：
- RAG 检索需求原文和历史用例
- Agent Wiki 提供测试策略、业务术语、风险规则

Web 执行：
- Agent Wiki 提供页面类型、组件语义、Skill 条件
- Memory 提供历史成功路径和 badcase

评测与回流：
- Eval Dataset 判断知识变更是否有效
- Badcase Feedback 更新 Wiki 或标记知识失效
```

### 6.6 知识更新与防污染

知识库必须支持治理，否则会变成新风险源。

每条知识需要：

```text
source：来源
version：版本
status：active / deprecated / suspicious
verified_by：被哪些 eval case 验证
last_verified_at：最后验证时间
owner：维护者
confidence：可信度
```

当 badcase 发现旧知识失效时：

```text
1. 标记旧知识为 suspicious
2. 加入 badcase
3. 修改 Wiki 卡片
4. 增加或更新 eval case
5. 跑回归
6. 通过后改为 active
```

## 7. 安全控制设计

### 7.1 为什么需要安全控制

Agent 最大风险不是“回答错”，而是：

* 调用了不该调用的工具；
* 修改了不该修改的数据；
* 访问了不该访问的页面；
* 提交了真实业务表单；
* 读取或泄露敏感信息；
* 执行了危险文件操作；
* 被 prompt injection 诱导越权。

OWASP 已将 LLM 应用安全作为独立方向维护，并提供针对生成式 AI 系统安全开发和治理的指南。([OWASP基金会][3]) OWASP 的 AI Agent 安全建议也明确强调工具最小权限、按工具设置权限范围、敏感操作显式授权等原则。([OWASP Cheat Sheet Series][4])

因此，本项目必须从第一版就设计 Safety Guard。

### 7.2 安全控制目标

```text
模型可以建议动作，但不能绕过系统边界直接执行
模型可以调用工具，但工具必须经过白名单、schema、权限和审计
模型可以生成测试任务，但危险动作需要人工确认
```

### 7.3 安全模块

```text
Safety Guard
├── Domain Allowlist
├── Tool Allowlist
├── Action Schema Validator
├── Parameter Validator
├── Risk Level
├── Human Approval
├── Dry-run Mode
├── File Guard
├── Sensitive Data Masking
└── Audit Log
```

### 7.4 sandbox_policy.yaml 示例

```yaml
network:
  allow_domains:
    - playwright.dev
    - docs.python.org
    - github.com
    - localhost
  block_private_ip: true

browser:
  max_steps: 15
  block_download: true
  block_upload: true
  block_payment: true

filesystem:
  writable_roots:
    - ./runs
    - ./reports
    - ./artifacts
  deny_patterns:
    - "/Users/*"
    - "/etc/*"
    - "C:\Users\*"

tools:
  search_page_text:
    risk_level: low
    side_effect: false
    approval_required: false

  write_eval_result:
    risk_level: low
    side_effect: true
    approval_required: false

  submit_real_form:
    risk_level: high
    side_effect: true
    approval_required: true

  delete_file:
    risk_level: high
    side_effect: true
    approval_required: true
```

## 8. 数据回流与持续优化设计

### 8.1 为什么必须做数据回流

如果失败只停留在日志里，系统不会变强。Agent Harness 必须把失败转为数据资产。

核心闭环：

```text
执行
→ 失败
→ Trace 记录
→ Badcase 分类
→ Root Cause 分析
→ Feedback Target
→ 修复 Context / Prompt / Tool / Skill / Assertion / Knowledge
→ 加入回归集
→ 再评测
```

### 8.2 Badcase 数据结构

```json
{
  "badcase_id": "badcase_20260429_001",
  "case_id": "fixture_gift_001",
  "run_id": "run_20260429_001",
  "task": "点击礼物入口并断言礼物面板出现",
  "error_type": "element_alias_error",
  "root_cause": "礼物入口是无文字 icon，context 中缺少语义 label",
  "feedback_target": "Agent Wiki / Alias Registry",
  "fix_plan": "新增 gift icon 的 component_alias 卡片，并加入 fixture_gift_001 回归集",
  "regression_added": true,
  "status": "fixed"
}
```

### 8.3 回流目标

| error_type               | feedback_target             | 修复方式                            |
| ------------------------ | --------------------------- | ------------------------------- |
| context_error            | Context Builder             | 调整元素过滤、增加 role/aria/title/class |
| element_alias_error      | Agent Wiki / Alias Registry | 增加组件语义卡片                        |
| planner_format_error     | Prompt / Schema             | 强化 JSON 输出约束，增加 repair          |
| planner_logic_error      | Prompt / Skill              | 增加任务拆解规则或 Skill                 |
| tool_param_error         | Tool Schema                 | 增加 enum、参数校验、默认值                |
| skill_precondition_error | Skill Registry              | 增加前置条件检查                        |
| executor_timeout         | Executor                    | 增加 wait/retry/visibility        |
| assertion_error          | Assertion Layer             | 增加多断言或调整断言时机                    |
| vision_assertion_error   | Vision Provider             | 增加区域截图、证据输出                     |
| knowledge_error          | Agent Wiki / RAG            | 标记知识失效、更新来源                     |
| safety_blocked           | Safety Policy               | 判断是正确拦截还是策略误伤                   |
| environment_error        | Fixture / Network           | 修复环境或标记不可用                      |

### 8.4 数据回流产物

```text
runs/<run_id>/trace.jsonl
runs/<run_id>/run_summary.json
reports/badcase_list.json
reports/feedback_summary.json
eval_cases/regression_cases.jsonl
knowledge/wiki_cards/*.json
```

### 8.5 优化是否有效的判断标准

不能只说“修了”。必须用指标判断：

```text
修复前 pass_rate
修复后 pass_rate
目标 error_type 是否下降
是否引入新 error_type
avg_latency 是否明显上升
json_valid_rate 是否下降
旧回归集是否通过
```

## 12. 最终面试表达

可以这样概括：

> 我做的不是普通 Browser Agent Demo，而是一个 Real-Web Agent Harness。它承接了我在 B 站 WebUI 自动化中的经验，但进一步把能力从“用例执行和脚本生成”扩展到“需求到测试任务生成、Agent 能力评测、Badcase 回流和安全控制”。系统会先采集真实 Web 页面或本地 fixture 的上下文，通过 Context Builder 和 Knowledge Layer 组织页面状态、业务术语和组件语义，再让 Planner 输出结构化 action。所有 action 都要经过 schema、安全策略和工具权限校验，执行过程由 Playwright 完成，并记录 Trace。最后通过 Evaluator 在固定 case 集上比较不同模型、Prompt、Context 和 Skill 的效果，失败样本会归因到 context、planner、executor、assertion、knowledge 或 safety，并回流到知识库、规则、Skill 和回归集里。这个项目的重点不是让 Agent 跑一次，而是让 Agent 在企业交付场景中可控、可测、可回放、可持续优化。
