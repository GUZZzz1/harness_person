````markdown
# Real-Web Agent Harness Quality Lab 总体计划书

## 1. 项目定位

本项目不是重新做一个普通 Browser Agent Demo，而是基于已有 WebUI 自动化 Agent 经验，抽象出一个可运行、可评测、可追踪、可回放、可持续优化的 Agent Harness。

项目核心目标：

> 在真实 Web 页面与本地可控页面上，构建一套 Agent 执行、上下文构造、结构化动作生成、Playwright 执行、Trace 记录、Evaluator 评测、Badcase 回流与模型服务可替换的最小工程闭环。

最终要服务两个目标：

1. 通过实操理解 Agent Harness、Context Engineering、Tool / Skill、Trace、Evaluator、Badcase Feedback 等模块；
2. 把项目过程中遇到的问题沉淀成可面试表达的“AI 应用工程化 + 测试开发质量保障”经验。

---

## 2. 当前机器与部署约束

### 2.1 MacBook Air

Mac 作为主控开发机。

职责：

- 项目主代码开发；
- Python / FastAPI / Playwright 运行环境；
- Browser Harness；
- Context Collector；
- Planner 客户端；
````

### GitHub 网络问题排查

如果 `git clone` 失败，先执行：

```bash
curl -I https://github.com --connect-timeout 10
```

如果终端不通但浏览器能打开 GitHub，优先检查代理：

```bash
git config --global --get http.proxy
git config --global --get https.proxy
```

```bash
curl -x http://127.0.0.1:7890 -I https://github.com --connect-timeout 10
```

如果能通，则配置 Git：

```bash
git config --global http.proxy http://127.0.0.1:7890
git config --global https.proxy http://127.0.0.1:7890
```

## 阶段 2：Fake Planner + Action Schema

### 目标

不接真实模型，先验证结构化动作、校验和执行链路。

### 任务

实现 Fake Planner，固定返回结构化动作：

```json
{
  "decision_type": "action",
  "action": "assert_text",
  "target_index": null,
  "params": {
    "text": "Locators"
  },
  "reason": "验证页面包含 Locators 文本",
  "expected": "页面存在 Locators"
}
```

实现 Pydantic Action Schema：

```text
decision_type: action / tool / skill / done
action: click / input / hover / wait_for_visible / assert_text / assert_dom
target_index
params
reason
expected
```

### 需要实现的文件

```text
backend/app/planner/schema.py
backend/app/planner/fake_planner.py
scripts/run_fake_planner_demo.py
```

### 验收标准

* Fake Planner 能返回合法 JSON；
* Pydantic schema 能校验成功；
* 非法 action 能被拒绝；
* 缺少必要字段能返回明确错误。

### 本阶段重点观察的问题

* 模型未来可能输出非法 JSON；
* action 类型需要 enum 限制；
* 参数缺失必须在执行前拦截；
* 不能让自然语言直接控制浏览器。

## 9. 面试沉淀方向

本项目最终要形成的面试表达不是：

> 我做了一个自动化浏览器 Agent。

而是：

> 我做了一个 Agent Harness，用于验证 Agent 在真实 Web 场景下的可靠性。系统通过 Context Collector 和 Context Builder 组织页面状态，通过 Planner 生成结构化动作，通过 Schema 校验和 Executor 执行动作，通过 Trace 记录每一步输入、输出和执行结果，再通过 Evaluator 批量评测不同 Context、Prompt、模型和 Skill 版本的效果。失败样本会被分类为 context、planner、executor、assertion、tool、skill 或 safety 问题，并回流到规则、评测集和后续优化中。
